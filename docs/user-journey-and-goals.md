# User Journey, Intents & System Goals

This document outlines the user personas, their mental models, specific user stories, and maps them directly down to the technical architecture, frontend, and backend implementation layers of the real-time Indonesian to English AI translator.

---

## 👥 1. User Roles & Mental Models

The system is designed around three primary user roles:

### 🎙️ The Speaker (Presenter / Pastor)
*   **Goal**: Broadcast low-latency translations of their spoken Indonesian words to the audience with zero hardware setup or technical friction.
   * **Mental Model**: *"I want to enter a simple PIN, tap 'Start', and speak naturally. My phone screen should stay awake, and a volume meter should reassure me that the microphone is active. When I finish, I tap 'Stop' and lock the screen."*
   * **User Stories**:
      * *As a speaker, I want to authenticate my session with a simple PIN so I don't have to manage complex accounts.*
      * *As a speaker, I want my phone screen to remain active (no timeout) during the broadcast so that microphone capture is never suspended by the OS.*
      * *As a speaker, I want to see a live volume meter so I can visually verify that my voice is being picked up.*
      * *As a speaker, I want the translation system to automatically fix spelling/speech errors and translate church/presentation terms contextually.*

### 📱 The Viewer (Congregation / Attendee)
*   **Goal**: Read live scrolling English translations of the presentation on their own device, with the option to listen in real-time.
   * **Mental Model**: *"I want to open a URL and immediately see the translation update. I shouldn't need to sign up or install anything. The interface must be dark so it doesn't glare in low-light environments, and I want to plug in my headphones to hear the speech in English."*
   * **User Stories**:
      * *As a viewer, I want to access the live translation by loading a web page without logging in.*
      * *As a viewer, I want a premium dark interface that eliminates screen glare for myself and people sitting next to me.*
      * *As a viewer, I want to adjust the text size easily so it is readable from any distance.*
      * *As a viewer, I want to toggle a text-to-speech option to listen to the translated English feed through headphones.*

### 🛠️ The Administrator / Developer (AV Tech)
*   **Goal**: Monitor the system's operational status and manage serverless resources at $0 infrastructure cost.
   * **Mental Model**: *"I want a debug panel to inspect raw voice transcripts side-by-side with translated blocks. I want the system to be stateless so it does not build up database storage costs, and it must defend itself from password brute-forcing automatically."*
   * **User Stories**:
      * *As an administrator, I want to run the system statelessly on serverless free tiers (Vercel & Supabase) to keep hosting costs at $0/month.*
      * *As an administrator, I want to monitor raw ASR inputs and translated outputs in real-time to debug translation fidelity and ASR accuracy.*
      * *As an administrator, I want failed PIN login attempts to trigger an automatic temporary IP lockout to protect the translation endpoint.*

## 🗺️ 2. Mapping Intents to System Architecture

Here is how the high-level user goals shape our technical architecture:

```
[User Intent] ──────────────────────────► [Architectural Solution]
1. $0 monthly infrastructure cost  ────► Stateless Supabase Realtime Channels (No database storage writes)
2. No-install audience access      ────► Single-Page Web Application hosted on Vercel
3. Keep-alive speaker recording   ────► Progressive Web App (PWA) + Screen Wake Lock API
4. Protect API secrets             ────► Edge Function token verification (PIN gate proxy)
5. Accurate, fluent translations   ────► Deepgram ASR streaming + DeepSeek sliding context window
```

## 💻 3. Frontend Implementation Layer

The user stories are realized on the frontend (`packages/frontend`) through dedicated components and routes:

*   **Viewer Console (`/`)**:
    *   Subscribes to the `sermon-live` Supabase Realtime channel to receive translation segments.
    *   Displays the teleprompter feed (latest translation highlighted at maximum size, preceding 2 segments dimmed at 50% and 25% opacity).
    *   Integrates browser-native `window.speechSynthesis` via **`TextToSpeechService`** to read translations aloud when headphones mode is toggled.
    *   Renders a floating accessibility bar allowing viewers to step font sizes up or down.
*   **Speaker Console (`/speaker`)**:
    *   Uses **`PinGate`** to prompt for access and stores credentials in `sessionStorage`.
    *   Invokes **`AudioOrchestrator`** to prompt for microphone access, request screen wake locks via the Screen Wake Lock API, and setup speech capture.
    *   Communicates with Deepgram via WebSockets (utilizing ephemeral tokens generated by the backend).
*   **Developer Debugger (`/admin`)**:
    *   Renders a running log of raw ASR inputs (Indonesian) alongside the translated outputs (English) to diagnose performance.

---

## ⚡ 4. Backend Implementation Layer

The backend services (`supabase`) handle security, LLM translation, and WebSocket dispatching:

*   **Deepgram Token Generator (`get-deepgram-token`)**:
    *   Authenticates incoming requests with the session PIN header.
    *   Calls Deepgram to issue a short-lived, ephemeral credential token, preventing exposure of the main master API keys on the frontend.
*   **Translation Brain (`translate`)**:
    *   Validates the PIN by comparing the SHA-256 hash of the input with server secrets (`ADMIN_PIN_HASH`).
    *   Implements rate limits (max 120 requests/minute per IP) and temporary IP lockouts (15 minutes after 5 failures) to protect backend endpoints.
    *   Takes the raw Indonesian segment, grabs the preceding 3 history segments, constructs the sliding context prompt, and calls DeepSeek.
    *   Parses the DeepSeek JSON response and broadcasts the segment to the `sermon-live` channel directly using administrative privileges.
